01

What a pilot would take

A first run is deliberately small — scoped to produce one replayable audit trail end to end, so the property being evaluated is the trail, not the wall-clock.

Scope

  • 1–2 member repos, one epic
  • one cross-repo contract in play, so blast-radius and the human gate actually fire

Prerequisites

  • Claude Code + the sdd-fleet plugin
  • a workspace vault over the repos; a service.json descriptor per repo
  • contract-registry and Jira access

What the bank staffs

  • gate approvers for epic/contract ratification and money / PII sign-off
  • an escalation owner for non-converging reviews

Success criteria

  • a replayable audit trail re-derived from .sdd/
  • zero un-gated money / PII merges
  • every acceptance criterion traced to a test before code
A pilot is one epic across one or two repos, run end to end. It succeeds or fails on the audit trail it leaves — not on how fast it ran.
02

Operating it — what we don't know yet

A candid list. These are the questions the pilot exists to answer — none of them has an honest number today, and we won't invent one.

Cost shape per change

  • the gates, the bounded review rounds, and the per-criterion checks cost wall-clock time and tokens
  • the shape of that cost — how much is fixed overhead per change, how much scales with the number of acceptance criteria — is to be measured in the pilot, per change and per epic

Approver load

  • the human gates concentrate on a few people: epic and contract ratification, money-movement and PII sign-off, escalation resolution
  • how many gate events one epic actually generates — and whether they queue on a single approver — is a pilot measurement, not an estimate

Failure modes

  • a bug in the harness itself fails closed — work blocks rather than slipping through; the pilot has to confirm that in practice and define the break-glass path
  • a model outage parks work safely at the last gated state
  • if an incident ships despite the gates, the replay trail is the investigation input — the gate set is then tuned, not trusted less

Rollout alongside existing CI/CD

  • sdd-fleet's gates act in-session, before a merge; the bank's existing CI/CD stays authoritative after it
  • how the two overlap without double-gating or gaps is worked out on the pilot repos, not assumed in advance
