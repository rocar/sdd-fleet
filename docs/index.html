sdd-fleet · a Claude Code plugin · a vision for regulated multi-repo delivery

Deterministic harness.
Probabilistic generation.

A spec-driven, multi-agent development system for a regulated, multi-repo estate — designed for audit and replay rather than for speed.

Harness

Deterministic & fail-closed

Gates, counting, routing, the loop. Reproducible, and not bypassable by an agent.

Model

Probabilistic judgement

Drafting, review, the one contested call. Isolated to a single logged role.

Human

Accountability

Ratification before work spreads to the repos; sign-off on money movement and PII. Accountability the machine cannot supply.

At a glance — the 30-second read

If you read nothing else

This section is the summary; the architecture pages carry the evidence; the one rule is the boundary.

What it produces

Per change, a gated, replayable audit trail: every acceptance criterion mapped to a test before code, every contract version-checked, every decision reconstructable from the record.

Where humans stay in control

Four points require a human signature — enforced by the harness, not by convention. They are the ◆ gates below.

epic + contract ratify ◆· money / PII sign-off ◆· blast-radius threshold ◆· escalation on non-convergence ◆
▣ What audit reads back

From artifacts that were gated, not asserted, the record supports one sentence per change: “checkout's step-up change was built against catalog snapshot vX, authorise@3.3.0, additive per ADR-217, human sign-off by Y, contract tests green on a write-locked suite.” The rest of this site explains how the harness guarantees that sentence — start with the one rule.

Replay means re-deriving that decision trail from the record — gates, inputs, counting, signatures — not regenerating identical code: generation is probabilistic; every consequence is identical on a re-run.

One change, end to end

One unit of work — Jira PAY-1843, "add an instant-payout endpoint" — traced through the per-repo machine. Every gate below is enforced by code; none depends on a reviewer remembering to check.

  1. SPEC — criteria are authored into .sdd/PAY-1843/; a testability floor refuses any criterion that isn't decidable.
  2. REVIEW — three role lenses fan out; an adversarial money-movement concern survives the survival vote and becomes a blocker.
  3. FINALIZE ▣ — fail-closed: SDD_FLEET_FINALIZE_REFUSE: {"code":2,"reasons":["open-blockers"]} until resolved, then SDD_FLEET_FINALIZE_PASS; no source before the flip.
  4. BUILD — every criterion maps to a failing test first, then the test paths are write-locked; code turns the locked suite green.
  5. CHANGE_REVIEW — blast radius computed from the derived catalog: 4 transitive consumers, money_movement.
  6. HANDOFF ◆ — either fact alone forces the human gate; the approval is pinned to the blast-radius signature (stale if the radius widens). SDD_FLEET_HANDOFF_APPROVE_PASS: {"feature":"PAY-1843","signature":"…"}
