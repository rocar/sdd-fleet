sdd-fleet · design concept · spec → system, governed

Four ideas.
One boundary.

sdd-fleet builds software from a spec across a regulated, multi-repo estate so that every consequential step — every merge gate, every review verdict, every contract publish — is reproducible code, not a model's judgment. The design is a deterministic harness: a control layer of plain code the AI runs inside. Code owns every consequence; the model judges only the narrow calls left over; a human owns the blast radius. It earns that guarantee by composing four proven ideas rather than inventing new ones — the provenance map is below.

01

The boundary — where code decides, where the model judges, where a human signs off

One line governs the whole system: every consequence lives in code; the model judges only the irreducible residue; a human owns the blast radius. Each layer is given exactly the authority it can be trusted with — and no more.

Harness owns every consequence

  • phase gates, the bounded review loop, survival-vote counting
  • semver + blast-radius computation, cross-repo dispatch
  • reproducible and fail-closed — identical on a re-run, and if anything is missing or unverifiable the gate blocks rather than allows

Model judges only the residue

  • the irreducible residue — the narrow, genuinely subjective calls no rule can settle
  • drafting a spec, per-criterion review, the single contested "is this diff breaking?" call
  • it never decides a consequence, and it never curates memory

Human owns the blast radius

  • the blast radius — how many dependent services a change reaches, and whether it touches money movement or PII
  • epic & contract ratification before fan-out; money / PII sign-off; escalation resolution
  • a human signature wherever the risk is too high to automate
Move a gate into a command's prose and you have moved a consequence into the model. So every consequence lives in a hook or a workflow — plain code, identical on a re-run.
02

What this buys a regulated estate

The boundary is not philosophy — it pays out as concrete properties an auditor, a risk officer, and an engineering lead each care about.

Replay

Re-derive, don't recollect

Every run reconstructs from the vault record — an auditor re-derives the decision instead of trusting anyone's memory of it.

Hard gates

Sign-off by computation

A change touching money-movement or PII data, or reaching N downstream consumers, forces a human gate — by code, not by a reviewer remembering to.

Multi-repo

Contracts hold the estate together

Published contracts are versioned and append-only; pinned consumers are stopped automatically on a breaking bump.

Precision

Auditable & replayable, not merely fast

Every acceptance criterion maps to a test before code; QA reads captured coverage output where tooling exists — never its own opinion of it.

Where humans stay in the loop. Four points require a human signature — enforced by the harness, not left to convention:
epic + contract ratify ◆· money / PII sign-off ◆· blast-radius threshold ◆· escalation on non-convergence ◆
03

One change, end to end

A single unit of work — Jira PAY-1843, "add an instant-payout endpoint" — moving through the per-repo machine. Every gate below is code, not a reviewer's diligence.

  1. SPEC — the spec and acceptance criteria are authored into the vault .sdd/PAY-1843/. A testability floor refuses any criterion that isn't decidable — no "works correctly", no TBD.
  2. REVIEW — three role lenses fan out and an adversarial pass flags a money-movement concern on the payout path. It survives the vote, so it becomes a blocker the loop must resolve.
  3. FINALIZE ▣ — a fail-closed gate refuses to freeze the spec while review blockers stand. SDD_FLEET_FINALIZE_REFUSE: {"code":2,"reasons":["open-blockers"]} → resolved, then SDD_FLEET_FINALIZE_PASS. No source can be written before this flip.
  4. BUILD — tests first: every criterion is mapped to a failing test (a fail-closed gate refuses source until the mapping is complete), then the test paths are write-locked so the oracle can't be edited to pass; code is generated to turn the locked suite green.
  5. CHANGE_REVIEW — blast radius is computed from the derived service catalog: this endpoint's contract has 4 pinned consumers, and the data carries money_movement.
  6. HANDOFF ◆ — either condition alone forces a human gate, regardless of the machine verdict. The fail-closed gate blocks the handoff until a human signs off, and the approval is pinned to the computed blast-radius signature — it goes stale if the radius widens. SDD_FLEET_HANDOFF_APPROVE_PASS: {"feature":"PAY-1843","signature":"…"}

05

Spec-driven development — what is true

Spec-driven development (SDD) inverts the code-first model: the specification, not the code, is the source of truth. The spec is "executable" in that an agent generates the implementation from it — so code becomes a continuously regenerated projection of the spec, not the primary artifact.

GitHub's Spec Kit frames the canonical loop as Specify → Plan → Tasks → Implement, each phase emitting a Markdown artifact that feeds the next: specify captures the what-and-why (deferring the tech stack), plan commits the how, tasks slices it into small reviewable units, implement executes. It suits AI agents because models complete patterns but cannot mind-read — a structured spec supplies the shared context a vague prompt cannot.

intent→ SPEC→ REVIEW→ FINALIZE ▣→ BUILD→ CHANGE_REVIEW→ HANDOFF ◆
what sdd-fleet takes The phase spine above is SDD's specify→plan→tasks→implement hardened into a state machine. The vault .sdd/ is the spec-as-source-of-truth: specs, acceptance, ADRs live there as the durable record; source is the projection. sdd-fleet adds what SDD leaves to convention — a fail-closed gate (▣) at each transition (no source before FINALIZED; no handoff on a failing or untraceable suite), so the spec doesn't just guide generation, it governs it.
06

Claude Code workflows — who holds the plan

Claude Code offers two registers of control. The defining difference is who holds the plan: with static primitives the model orchestrates turn by turn; a dynamic workflow moves the loop into code.

Static the rails

  • fixed entry points — work starts only where the system allows, never mid-stream (slash commands)
  • tool-level gates that fail closed at the boundary — e.g. no source written before the spec is finalized (hooks)
  • the rulebook the model loads — the protocol and the review rubric (skills)
  • the role lenses — architect · coder · QA, plus the sole writer of record (subagents)

Dynamic the engines

  • model-authored code that an isolated runtime executes in the background
  • the script holds the loop, the branching, and the intermediate results — not the chat context
  • deterministic and re-runnable; only the work inside each model call is model-powered
  • purpose-built engines for spec review, change review, and build
what sdd-fleet takes Both registers, deliberately. Static primitives are the rails — the commands, the fail-closed hooks, the skills — that a model can never bypass. Dynamic workflows are the fan-out engines — they hold the bounded review loop, the survival vote, and the state writes as code, so the consequence is identical on a re-run. The boundary the whole system rests on (consequence in code, not prose) is exactly Claude Code's own dynamic-vs-static distinction, applied as a design rule.
07

The deep-research verify pattern — what survives

Deep-research agents establish a fact by adversarial verification: fan out to gather claims, then have independent skeptics try to refute each one, and let a vote — not a single model's say-so — decide what survives.

deep-research · skeptical of the claims

  • a claim is guilty until proven innocent
  • if a skeptic is uncertain it defaults to refuted; if all abstain, the claim is killed
  • goal: precision — keep only what verifies (protect the report from false claims)

sdd-fleet · skeptical of the work

  • a concern is innocent until proven guilty
  • it survives unless refuted by a different role with a resolving citation, judged sound by one neutral adjudicator
  • goal: recall — a guaranteed floor on what gets caught (never miss a real defect)
what sdd-fleet takes — and flips The machinery is identical (fan-out → adversarial challenge → code-level vote), but the burden of proof is inverted. And the vote here carries a consequence: a surviving blocker doesn't just drop from a report — it blocks the merge, the loop is bounded (≤3, count-must-fall), and it escalates to a human. Verification becomes a control gate, not a quality filter.
08

Obsidian's link model — how memory holds

In Obsidian, you author forward links in your notes; the backlink pane and the graph view are derived — computed from those authored links, never hand-kept. The record is what you wrote; the index is a projection of it.

what sdd-fleet takes The vault is one Obsidian vault over the member repos, so memory is plain-markdown forward links — reviewed in the PR, diffable, as auditable as the prose around them — while the backlink graph is a derived projection a human and the agent recover identically. The discipline that falls out: standard markdown links, no [[wikilinks]] (they die on GitHub and to the gate parsers), cross-tree refs by stable ID, and a semantic index may sit on top for discovery but never feeds a gate. Memory is authored, never model-curated.

The synthesis

Each pillar supplies one face of the same boundary. Together they make a system that is auditable and replayable, not merely fast.

Deterministic where it counts, model-judged only at the residue, human-owned at the blast radius — assembled from four ideas that each already drew one edge of that line.
Operating tradeoffs, honestly. This optimises for audit and replay, not raw speed — the gates, the bounded review loop, and the per-criterion checks cost wall-clock and tokens. The human gates concentrate load on a few approvers (epic / contract ratification, money / PII sign-off), and most of the engineering effort shifts up-front into the spec and its acceptance tests. The payoff is a change you can re-derive and defend, not one you produced fastest.

What a pilot would take

A first run is deliberately small — scoped to produce one replayable audit trail end to end, so the property being evaluated is the trail, not the wall-clock.

Scope

  • 1–2 member repos, one epic
  • one cross-repo contract in play, so blast-radius and the human gate actually fire

Prerequisites

  • Claude Code + the sdd-fleet plugin
  • a workspace vault over the repos; a service.json descriptor per repo
  • contract-registry and Jira access

What the bank staffs

  • gate approvers for epic/contract ratification and money / PII sign-off
  • an escalation owner for non-converging reviews

Success criteria

  • a replayable audit trail re-derived from .sdd/
  • zero un-gated money / PII merges
  • every acceptance criterion traced to a test before code
The ask: pick one epic in one or two repos and run it end to end — then measure the audit trail, not the speed.
