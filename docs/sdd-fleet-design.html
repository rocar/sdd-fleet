What it is

A spec-driven, multi-agent build system where plain code — not the model — owns every consequence: which review concerns block a merge, when a gate opens, when work escalates to a human.

What it produces

Per change, a gated, replayable audit trail: every acceptance criterion mapped to a test before code, every contract version-checked, every decision reconstructable from the record.

Where humans stay in control

Contract & epic ratification before any work spreads to the repos, money-movement / PII sign-off, and escalation when review won't converge — enforced by the harness, not by convention.

The tradeoff, stated plainly

Optimised for audit and correctness, not raw speed. Decisive review is front-loaded onto the contract shape up front, rather than spent per-PR after the fact.

Maturity & evidence

This is a real, tested Claude Code plugin, not a slide: every fail-closed gate ships with a failing-test-first suite, run in CI on macOS (bash 3.2) and Linux. A pilot would validate it on a live contract change; the mechanics it rests on are built today.

▣ What audit reads back — every change, every time

From artifacts that were gated, not asserted, the trail reconstructs one sentence without anyone reconstructing it from memory: “checkout's step-up change was built against catalog snapshot v­X, authorise@3.3.0, additive per ADR-217, human sign-off by Y, contract tests green on a write-locked suite.” How the harness guarantees that sentence is what the rest of this brief explains.

00

Why the scaffolding exists

An AI agent generates plausible code non-deterministically — re-run the same prompt, get a different answer — and left unbounded it also decides its own consequences: whether a concern matters, whether to merge. A regulated bank cannot ship that: when a wrong call moves money or leaks PII, “it usually gets it right” is not a control, and nothing is reproducible or auditable. The fix is not a better model — it is a deterministic harness over probabilistic generation.

Every input that could vary run-to-run — what a reviewer flags, how code is written — is the model's. Every consequence drawn from those outputs — does a concern survive, does a gate open, does this escalate — is the harness: pure code, identical on a re-run. The model judges; the harness decides what happens next. The residue left to the model is the narrow set of subjective calls no rule can settle — a draft, a review judgement, one contested decision. That single boundary is what makes the system auditable, and it is drawn — by colour — at every step that follows.

Deterministic where it counts, model-judged only at the residue, human-owned at the blast radius.
01

The architecture: a control stack and its records

Two halves: a stack of three control layers — who acts — over the record planes where truth lives. At the centre is the workspace vault, the single source of truth that Jira, the registry and the catalog orbit. Each store owns exactly one kind of fact — because the classic failure is two stores claiming the same truth.

Layer 3 · workspace

The workspace — plan above, pull below Model Harness Human

A parent folder over the member repos — git submodules, one Obsidian vault. epic-plan (model + human) authors the dependency DAG and the contract design into workspace/.sdd/, ratified before any work spreads to the repos. A developer then pulls the next ready story with next-story — a deterministic resolver over live status + the registry — and starts it. The conductor is the optional autonomous variant of that same resolver (a level-triggered reconciler, no model in the loop) — so the widest-blast-radius layer is the most deterministic, not the least. Planning is a command; dispatch is deterministic plumbing.

Layer 2 · per-repo

The feature machine — one per repository Model Harness

SPEC → REVIEW → FINALIZE → BUILD → CHANGE REVIEW → HANDOFF. A coded harness runs the phases and routes the work; the model only drafts and reviews within them, never writing source or reordering a phase. Single-worktree, so two changes never race in one repo.

Layer 1 · enforcement

Hooks — the gates are the product Harness

The harness owns the sequence; the hooks own the boundary. They are fail-closed — a gate's default is to block, and opening it takes an explicit, coded allow, so a missing or unverifiable check stops the action rather than letting it slip through. They block a forbidden action however it is initiated — no source before the spec is finalized, no handoff on a failing or untraceable suite, no contract change without a published, version-bumped spec — catching any deviation the harness did not foresee, the model going off-script inside a phase included. Why Claude Code specifically: this is in-session, tool-level interception — a hook that blocks a forbidden write before it happens, not a review comment after the fact — plus review delegated to separate subagent lenses the model cannot overrule. A suggestions-only assistant cannot do that.

The vault — two .sdd/ levels
One Obsidian vault over the submodules, but two .sdd/ levels: workspace .sdd/ (epic plans, contract design, promoted lessons) and each repo's own .sdd/ (that repo's specs, acceptance, ADRs). Each fact lives at exactly one level.

In a bank's terms. These planes map onto controls a regulator already recognises: the replay trail is examiner-ready change-control evidence; ratification and the HANDOFF gate are segregation-of-duties / four-eyes on money movement; one fact, one store means no conflicting systems of record to reconcile under audit. And the planes are likely tools you already run — Jira, a contract registry, a Backstage-style catalog, and a git-backed document store.

02

The workspace

A bank change spans services, and the dangerous part is the interface between them. The workspace is where that change is planned and driven — a parent folder with the member repos as git submodules and a single Obsidian vault over them, an orchestration layer, not a monorepo. Two ideas make the interface safe: the contract is the shared spec, and a computed catalog gives every change its neighbourhood.

Because the member repos are git submodules pinning exact SHAs, the workspace at a commit is a reproducible snapshot of the whole estate. A hook refuses to spec any story whose epic is not yet ratified — so the ratified contract design is the context every repo inherits.

The contract is the shared spec

The schema — OpenAPI, Avro, proto — is the cross-service spec, and consumer-driven contract tests are its acceptance. A producer's change must not break a consumer's registered expectation; that test is the gate. Tests-first, extended across repos.

Structural awareness is computed, not curated

Each repo carries one small, gated descriptor — identity, consume edges, lifecycle, data classes. The system map falls out of every descriptor and published contract: you maintain per-service metadata; the graph regenerates. Don't build an awareness store — consume the catalog (Backstage or the minimal equivalent).

# service.yaml — repo root · committed · gated · human-owned
id: payments-authoriser        # stable, never reused
team: payments-core
lifecycle: active              # active | deprecating | migrating
data_classes: [pii, money_movement]   # feeds blast-radius
produces:
  - contract: payments.authorise
    version: 3.3.0
    spec: contracts/authorise.openapi.yaml
consumes:                       # this edge builds the graph
  - contract: ledger.post     version: ">=2.1 <3"
  - contract: fraud.score    version: ">=1.4 <2"

The catalog feeds the gates, not just the prompt. Reverse edges make blast radius deterministic: a contract change resolves to its N transitive consumers — computed, not guessed. So the human-gate rule is principled — “reaches N consumers, or a money-movement service” — not a hardcoded “touches auth.” A semver check and pinned-consumer lookup (both code) leave the model one call: is this diff breaking beyond its version bump?

The conductor — autonomous dispatch optional

The dispatch above is a pull: a developer runs next-story and starts the ready story. The conductor is the optional autonomous variant of that same resolver — a level-triggered reconciler (scripts only, no command, no model) that drives the frontier hands-off across a large estate. Each tick reads ground truth — status + the registry — fresh, signals the ready set, and holds no private state, so a crash mid-tick is invisible: on restart it resumes from real state. It never invokes the per-repo machine — it only signals — so pull and autonomous share the exact same ready-frontier core; the only difference is who pulls the trigger. See ADR-0001 for the execution & deployment model.

One writer per unit of work. One orchestrator per repo worktree, one conductor per epic — so two runs never race on the same state as the estate scales.
03

Inside one repository

The same three-lens method runs at every stage, so design rigour, implementation and quality are weighed throughout — not just at review. The lens is the unit; the machine carries one change from intent to merge.

The four phases, fully expanded for implementers
Architect

“Does this fit conventions and the wider system?”

  • Interfaces & contracts
  • System & blast-radius impact
  • Alignment to existing patterns
Coder

“Is the code correct, idiomatic, maintainable?”

  • Correctness & edge handling
  • Readable implementation
  • Docs kept in step
QA

“Is it verified, and does it hold under stress?”

  • Coverage & gaps (from real runs)
  • Acceptance-criteria mapping
  • Risk & regression

Two hardening rules that carry the weight

A floor under detection. The machine is good at refusing a bad concern, but nothing guarantees a real one gets raised — and three lenses of one model share blind spots. So each lens returns an explicit pass / fail / concern on every criterion (silence becomes impossible), a dedicated adversarial pass hunts security, money and PII, and QA is grounded in real coverage output — not the model's opinion of it.

A trustworthy oracle. Everything rests on the test suite, and the cheapest route to green is weakening the test. Two code-enforced fixes: every criterion maps to a test before implementation, and the test paths are write-locked for the run — the coder physically cannot edit them.

The survival vote is where the line is drawn most sharply: code checks what code can check; one neutral model judges the rest.

Filter · different role Harness

The refuter is never the concern's own author. No echo chamber.

Filter · citation resolves Harness

The cited line must exist in the spec or contract. No citation, discarded.

Adjudicator · is it sound? Model

One stake-free role rules whether the reasoning holds and the cited line truly supports it. Verdict logged. Replaces the char-count that only pretended to judge soundness.

The loop is bounded and regression-guarded. Surviving blockers route back for up to three rounds; the full suite re-runs each round; a blocker's identity is a hash of its mapped criterion, so “same blocker” is a deterministic comparison; if the count doesn't strictly fall, it escalates early rather than converging on something worse. After three rounds, the machine stops guessing and pulls in a person.
04

Memory: a linked vault, not a curated store

The estate must remember its plans, specs, decisions and lessons — and the trap is letting a model curate that memory, putting non-determinism in the foundation the next spec builds on. The fix falls out of the structure already chosen: one Obsidian vault, two .sdd/ levels (workspace plans above, each repo's specs and ADRs below), all plain markdown — so a human and Claude Code open the identical files.

The record — authored markdown & links Harness-disciplined

Epic plans and contract designs in the workspace .sdd/; specs, acceptance and per-change ADRs in each repo's own .sdd/; cross-service lessons promoted up from repo to workspace level — all gated, git-versioned, changed only through review. Documents correlate through links written into the markdown, and because the repos are submodules of the one vault, those links cross repo boundaries. Each link is part of the record: reviewed in the PR, diffable, as auditable as the prose around it.

The index — derived backlinks & graph Derived · deterministic

Obsidian builds the backlink pane and the graph view from those authored links — nothing hand-kept, and it regenerates exactly from the record. Claude Code recovers the same neighbourhood by reading the links directly. Navigation is bidirectional, spans the whole workspace, and is a projection of recorded fact — never a model's recollection.

# workspace/   — ONE Obsidian vault · member repos are git submodules (pinned SHAs)
#   .sdd/                      ← WORKSPACE level · epic plans · contract design · lessons
#     epic-PAY-1842.md         plan + contract design — links DOWN into the repos
#   authoriser/                submodule @a1b2  (a member repo)
#     .sdd/                    ← REPO level · this repo's spec · acceptance · ADRs
#       spec.md  acceptance.md  ADR-217.md
#   checkout/   ledger/        …each its own submodule, each with its own .sdd/

# authoriser/.sdd/spec.md
Implements [PAY-1843](https://jira/PAY-1843) · contract payments.authorise@3.3.0
See [acceptance](./acceptance.md)
# in-repo links are relative; refs outside this repo use a stable ID, never ../../
# backlinks are computed from these — never written by hand
Forward links are the record; backlinks are the index derived from them — authored fact, deterministic projection, the same split drawn everywhere.
Memory stays out of the model's hands. The vault is plain, diffable, git-versioned markdown that a human and Claude Code read identically; no model curates it, and a semantic index may sit on top for discovery but never feeds a gate.
05

What this buys, and what you can replay

Every decision in the system falls into one of three columns — and that division is the governance story, not a footnote to it.

Harness owns the consequence

  • Phase gates & the bounded loop
  • Survival-vote counting & routing
  • Semver & pinned-consumer checks
  • Blast-radius computation
  • Workspace dispatch & reconciliation

Model judges the residue

  • Drafting the plan, spec, tests, code
  • Per-criterion review & the adversarial pass
  • One soundness verdict per contested refutation
  • “Is this diff semantically breaking?”
  • Optional semantic index (regenerable)

Human owns the blast radius

  • Epic ratification, before work spreads to the repos
  • Contract-design sign-off
  • Money-movement & PII gates
  • Escalation resolution
  • Promoting a lesson into the record
