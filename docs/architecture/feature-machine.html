Inside one repository

One state machine carries one change from intent to merge, inside a single repository.

The four phases, fully expanded for implementers
Architect

“Does this fit conventions and the wider system?”

  • Interfaces & contracts
  • System & blast-radius impact
  • Alignment to existing patterns
Coder

“Is the code correct, idiomatic, maintainable?”

  • Correctness & edge handling
  • Readable implementation
  • Docs kept in step
QA

“Is it verified, and does it hold under stress?”

  • Coverage & gaps (from real runs)
  • Acceptance-criteria mapping
  • Risk & regression

Two hardening rules

A floor under detection

  • every lens returns an explicit pass / fail / concern on every criterion — silence is impossible
  • a dedicated adversarial pass hunts security, money and PII
  • QA is grounded in captured coverage output; where no tooling exists the fallback is explicitly labelled a model assessment, never passed off as measurement

A trustworthy oracle

  • every acceptance criterion maps to a test before implementation
  • test paths are write-locked for the run — the coder physically cannot edit them
The survival vote is where the line is drawn most sharply: code checks what code can check; one neutral model judges the rest.

Filter · different role Harness

The refuter is never the concern's own author.

Filter · citation resolves Harness

The refutation must cite its artifact, and code verifies the cited quote actually appears there. A refutation without one is discarded.

Adjudicator · is it sound? Model

One stake-free role rules whether the reasoning holds and the cited line truly supports it, and the verdict is logged.
