01

A linked vault, not a curated store

Don't let a model curate memory — that puts randomness under the next spec. Instead: one Obsidian vault of plain markdown, two levels — workspace plans above, each repo's specs below. A human and Claude Code open the same files.

workspace/                        ← one vault; member repos are git submodules
├─ .sdd/                          ← workspace level · epic plan · contracts · lessons
└─ authoriser/                    ← a member repo
   └─ .sdd/
      └─ PAY-1843/                ← one folder per story
         ├─ spec.md               Implements [PAY-1843] · payments.authorise@3.3.0
         ├─ acceptance.md         ← linked from spec.md as [acceptance](./acceptance.md)
         └─ DECISIONS.md          ← the append-only decision log
# backlinks are computed by the vault from these links — never written by hand, vault-only
the rules Standard markdown links only — no [[wikilinks]] (they break on GitHub and to the gates). Cross-repo references use stable IDs. A search index may sit on top for discovery but never feeds a gate. These rules are borrowed from Obsidian's link model; the rest of the design's lineage: Three ideas, one boundary.
02

The three documents

Specs, decisions, lessons. The structure is not a convention: specs are checked on every write, decision logs are append-only, and a lesson needs a human to get in.

The spec. One per feature. A STATUS line plus eight fixed sections — a write missing any of them is rejected by a hook. The hook checks structure only; whether the content is good is what review is for.

# repo/.sdd/PAY-1843/spec.md
STATUS: DRAFT            DRAFT → IN_REVIEW → FINALIZED
# Instant payout endpoint
## Overview              the problem, the user, why now
## Goals                 outcomes, not implementation
## Non-goals             what's out of scope — an empty one is a [blocker]
## Behavior              concrete enough to build from without guessing
## Interfaces / Contracts  API shapes · schemas · error envelopes
## Constraints           performance · security · regulatory
## Risks                 naming one is not a failure; hiding one is
## Acceptance Criteria   See [acceptance.md](./acceptance.md)

The decision (ADR). Every decision that survives review gets an entry. Entries are never edited — to change course, add a new one that supersedes the old. A decision made silently isn't a decision yet.

# repo/.sdd/PAY-1843/DECISIONS.md — append-only
## ADR-7: Route step-up auth through the existing risk engine
- Date: 2026-07-04
- Status: accepted          or: superseded by ADR-9 · deprecated
Context · Decision · Consequences — a few lines each.
# product-wide decisions live in .sdd/_product/DECISIONS.md — cited as "product ADR-3"

The lesson. A finding that matters beyond its repo. A human promotes it up to the workspace; the model can propose, never promote.

# workspace/.sdd/_epic/lessons/idempotency-keys.md — promoted by a human ◆
Consumers retry on gateway timeout. Every money-movement endpoint needs an
idempotency key in the contract, not in the implementation.
Evidence: [PAY-1843] · payments.authorise@3.3.0
03

How they link

Four kinds of link. A hook checks every write against them.

Inside a feature — relative

  • [acceptance](./acceptance.md)
  • same folder, so the link moves with the feature

Across repos — stable ID

  • [PAY-1843] · payments.authorise@3.3.0
  • never ../../ — it works in the vault but breaks in the repo's PR view; an ID works anywhere

Between decisions — supersession

  • Status: superseded by ADR-9
  • old entries never change — the chain is the history

Between scopes — qualified

  • product ADR-3
  • keeps feature and product numbering apart
Adopting an existing repo? A one-time sweep runs every .sdd/ file through the same rules and lists what the gate would block — fix the list, re-run until clean.
04

How it grows — and comes back

.sdd/ is committed to git, so the record grows on its own: one folder per feature, decisions append-only. The hard part is the way back — getting what accumulated in front of the model again. Today exactly one return path is automated.

05

Finding things when the vault grows

A hundred features in: a hundred folders, hundreds of decisions. Nobody reads everything. What gets memory back out — honestly.

What works today

  • the CLAUDE.md block — small, always loaded, only settled facts
  • grep by stable ID — every cross-repo reference is exact-match findable
  • the vault graph — for humans browsing a neighbourhood
  • a search index — allowed for discovery, not built, never a gate

Still open

  • nothing forces reading lessons/ when a new spec is written — lessons go up, but nothing brings them back
  • product decisions grow forever, and every feature inherits them all — at some size that needs its own digest
  • no way to know which old decision matters to a new spec — today, a person remembers
Gaps in the protocol, not the tooling. Each has an obvious fix — a spec scaffold that cites the lessons it read, a digest of product decisions, a search index that suggests but never gates. Until built, they are stated limits, not hidden ones.
