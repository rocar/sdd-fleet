01

Three borrowed ideas

None of this is new. Three existing ideas each answer one question — what is true, who holds the plan, what survives review — and together they draw one rule.

02

Spec-driven development — what is true

The spec, not the code, is the source of truth. An agent generates the code from the spec, so code can be regenerated at any time — the spec is the thing you keep correct.

what sdd-fleet takes The spec-first order, hardened into a state machine. Specs, acceptance criteria, and decisions live in .sdd/ as the durable record; source code is generated from them. What spec-driven tools leave to habit, sdd-fleet enforces — a fail-closed gate ▣ at every step.
03

Claude Code workflows — who holds the plan

Claude Code gives two levels of control, and the difference is who holds the plan: with prompts and commands, the model steers turn by turn; a dynamic workflow moves the loop into code.

what sdd-fleet takes Both, deliberately. Commands, hooks, and skills are the rails a model cannot bypass. Dynamic workflows run the heavy parts — the bounded review loop and the survival vote, where the counting is code. State changes go through code too, so a re-run gives the same consequence every time.
04

The deep-research verify pattern — what survives

Deep-research agents check a claim by attacking it: gather claims, send independent skeptics to knock each one down, and let a vote decide what survives — not one model's say-so.

what sdd-fleet takes — and flips Same machinery, flipped burden of proof: a concern stands unless it is refuted with evidence. And the vote has teeth — a surviving blocker blocks the merge.
