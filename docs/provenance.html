sdd-fleet builds software from a spec across a regulated, multi-repo estate. Its design rule is short: every consequential step — a merge gate, a review verdict, a contract publish — is executed by reproducible code, never decided by a model. The model drafts and reviews inside that harness; a human signs off where the blast radius is high. None of the ideas underneath is new. This page traces where each of the four comes from, and what the design takes from each.

02

Spec-driven development — what is true

Spec-driven development (SDD) inverts the code-first model: the specification, not the code, is the source of truth. The spec is "executable" in that an agent generates the implementation from it, so code becomes a continuously regenerated projection of the spec rather than the primary artifact.

GitHub's Spec Kit frames the canonical loop as Specify → Plan → Tasks → Implement, each phase emitting a Markdown artifact that feeds the next: specify captures the what-and-why (deferring the tech stack), plan commits the how, tasks slices it into small reviewable units, implement executes. It suits AI agents because models complete patterns but cannot mind-read — a structured spec supplies the shared context a vague prompt cannot.

intent→ SPEC→ REVIEW→ FINALIZE ▣→ BUILD→ CHANGE_REVIEW→ HANDOFF ◆
what sdd-fleet takes The phase spine above is SDD's specify→plan→tasks→implement hardened into a state machine. The vault .sdd/ is the spec-as-source-of-truth: specs, acceptance, ADRs live there as the durable record; source is the projection. sdd-fleet adds what SDD leaves to convention: a fail-closed gate (▣) — the default is to block; opening takes an explicit, coded allow — at each transition: no source before FINALIZED, no handoff on a failing or untraceable suite. The transitions are enforced, not followed by habit.
03

Claude Code workflows — who holds the plan

Claude Code offers two registers of control. The defining difference is who holds the plan: with static primitives the model orchestrates turn by turn; a dynamic workflow moves the loop into code.

Static the rails

  • fixed entry points — work starts only where the system allows, never mid-stream (slash commands)
  • tool-level gates that fail closed at the boundary — e.g. no source written before the spec is finalized (hooks)
  • the rulebook the model loads — the protocol and the review rubric (skills)
  • the role lenses — architect · coder · QA, plus the sole writer of record (subagents)

Dynamic the engines

  • model-authored code that an isolated runtime executes in the background
  • the script holds the loop, the branching, and the intermediate results — not the chat context
  • deterministic and re-runnable; only the work inside each model call is model-powered
  • purpose-built engines for spec review, change review, and build
what sdd-fleet takes Both registers, deliberately. Static primitives are the rails — the commands, the fail-closed hooks, the skills — that a model cannot bypass. Dynamic workflows are the fan-out engines: they hold the bounded review loop, the survival vote — a concern stands unless refuted with evidence, and the counting is code — and the state writes as code, so the consequence is identical on a re-run. The boundary the whole system rests on (consequence in code, not prose) is exactly Claude Code's own dynamic-vs-static distinction, applied as a design rule.
04

The deep-research verify pattern — what survives

Deep-research agents establish a fact by adversarial verification: fan out to gather claims, have independent skeptics try to refute each one, and let a vote — not a single model's say-so — decide what survives.

deep-research · skeptical of the claims

  • a claim is guilty until proven innocent
  • if a skeptic is uncertain it defaults to refuted; if all abstain, the claim is killed
  • goal: precision — keep only what verifies (protect the report from false claims)

sdd-fleet · skeptical of the work

  • a concern is innocent until proven guilty
  • it survives unless refuted by a different role with a resolving citation, judged sound by one neutral adjudicator
  • goal: recall — a guaranteed floor on what gets caught (never miss a real defect)
what sdd-fleet takes — and flips The machinery is identical — fan-out, adversarial challenge, code-level vote — but the burden of proof is inverted: a concern survives unless refuted with evidence. And here the vote carries a consequence. A surviving blocker blocks the merge; the loop is bounded to three rounds, and the blocker count must strictly fall or the run escalates to a human. Verification is used as a control gate rather than a quality filter.

06

The synthesis

Each pillar supplies one face of the same boundary. Together they make a system that is auditable and replayable end to end.

PillarContributesTo the boundary
Spec-driventhe spec as source of truth; the phase spinefixes the deterministic record the model generates against
Claude Code workflowsstatic rails + dynamic engines; "plan in code"makes every consequence code; only the residue — the judgment calls no rule can settle — is model
Deep-research verifyfan-out → refute → vote (inverted default)turns review into a fail-closed gate, escalating to the human
Obsidian linksauthored forward links; derived backlinks/graphkeeps memory a deterministic projection, never model-curated
Each of the four ideas had already drawn one edge of this line. sdd-fleet assembles them into a single boundary: deterministic where it counts, model-judged at the residue, human-owned at the blast radius.
Operating tradeoffs. The design optimises for audit and replay, not raw speed. The gates, the bounded review loop, and the per-criterion checks cost wall-clock time and tokens. The human gates concentrate load on a few approvers — epic and contract ratification, money and PII sign-off. Most engineering effort moves up-front, into the spec and its acceptance tests. In return, every change can be re-derived and defended after the fact.
