01

Provenance — the four ideas we composed

None of the ideas underneath is new. Four existing ideas each draw one edge of the same boundary — what is true, who holds the plan, what survives review, and how memory holds.

02

Spec-driven development — what is true

Spec-driven development (SDD) inverts the code-first model: the specification, not the code, is the source of truth. The spec is "executable" in that an agent generates the implementation from it, so code becomes a continuously regenerated projection of the spec rather than the primary artifact.

what sdd-fleet takes The phase spine is SDD's specify→plan→tasks→implement hardened into a state machine. The vault .sdd/ is the spec-as-source-of-truth: specs, acceptance, ADRs live there as the durable record; source is the projection. sdd-fleet adds what SDD leaves to convention: a fail-closed gate (▣) at each transition — the transitions are enforced, not followed by habit.
03

Claude Code workflows — who holds the plan

Claude Code offers two registers of control. The defining difference is who holds the plan: with static primitives the model orchestrates turn by turn; a dynamic workflow moves the loop into code.

what sdd-fleet takes Both registers, deliberately. Static primitives are the rails — the commands, the fail-closed hooks, the skills — that a model cannot bypass. Dynamic workflows are the fan-out engines: they hold the bounded review loop, the survival vote — a concern stands unless refuted with evidence, and the counting is code — and the state writes as code, so the consequence is identical on a re-run. The boundary the whole system rests on (consequence in code, not prose) is exactly Claude Code's own dynamic-vs-static distinction, applied as a design rule.
04

The deep-research verify pattern — what survives

Deep-research agents establish a fact by adversarial verification: fan out to gather claims, have independent skeptics try to refute each one, and let a vote — not a single model's say-so — decide what survives.

what sdd-fleet takes — and flips The machinery is identical — fan-out, adversarial challenge, code-level vote — but the burden of proof is inverted: a concern survives unless refuted with evidence. And here the vote carries a consequence — a surviving blocker blocks the merge. Verification is used as a control gate rather than a quality filter.
05

Obsidian's link model — how memory holds

Authored forward links are the record; backlinks and the graph are a derived projection — never hand-kept, never model-curated. The full mechanics, and what sdd-fleet takes, live on the memory page.
